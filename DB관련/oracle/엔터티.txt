SQLD 

데이터 모델링 : 1. 현실세계를 데이터베이스에 추상화 하는 것.
	        2. 데이터 모델링은 의사소통을 통해 고객의 업무를 이해하고 데이터 모델링 표기법을 통해 모델링 한다.
	        3. 고객의 비지니스 모델을 데이터 모델로 표현한다.

데이터 모델링의 특징  : 추상화 - 현실세계를 표현하고 정의한다.
		      명확화 - 업무적으로 의미를 가져야한다.
		      단순화 - 누구나 쉽게 이해가 가능해야한다.

데이터 모델링 단계 : 1.의사소통 단계
		  2. 개념적 모델링 단계 
			- E-R 모델링
			- DBMS 에 독립적인 개념적 스키마 모델링 단계
			- 추상화 수준이 가장 높은 단계
			- 전사적 관점에서 기업의 데이터를 모델링한다.
			- 업무적인 측면에서 모델링한다. 
		  3. 논리적 모델링 단계
		  	- 특정 DBMS에 종속된 논리적 데이터 모델링 단계
			- 정규화 과정을 통해 재 사용률을 높인다.
			- 식별자 관계 속성등을 정의한다.
		  4. 물리적 모델링 단계
		  	- 성능과 보안 가용성을 고려하여 데이터 베이스의 구조를 모델링 하는 단계 
		  5. 데이터베이스 구현 단계
			- DDL을 이용하여 DB를 생성한다.

데이터 모델링의 관점 - 1. 데이터 : 업무에 필요한 데이터를 중심으로 모델링 (구조 분석, 정적분석)
		   - 2. 프로세스 : 업무를 진행하는 작업을 중심으로 모델링 (시나리오 분석, 동적분석, 도메인 분석)
		   - 3. 데이터와 프로세스 : 프로세스와 데이터와의 관계를 의미한다. (CRUD : Create Reletion Update Delete 분석)

데이터 모델링을 위한 ERD 절차 - 1. 엔터티를 돌출하고 그린다.
			   - 2. 엔터티를 배치한다.(주요 엔터티를 좌측 상단에 배치)
			   - 3. 엔터티 간의 관계를 설정한다.
			   - 4. 관계명을 서술한다.
			   - 5. 엔터티간의 관계 참여도를 작성한다.
			   - 6. 엔터티간의 관계 필수여부를 작성한다.


데이터 모델링시 고려사항 : 1. 데이터모델의 독립성
		          2. 고객의 요구사항 표현
		          3. 데이터 품질 확보


3층 스키마 :  필요성 - 1. 사용자, 설계자, 계발자가 데이터베이스를 바라보는 관점에 따라 데이터베이스를 기술하고 이들간의 관계를 정의
		    2. 데이터베이스의 독립성을 확보하기 위한 방법
		    

3층 스키마의 독립성 : 1.물리적 독립성 - 저장구조가 바뀌더라도 응용 프로그램 및 개념스키마의 영향이 없어야 한다.
		   2. 논리적 독립성 - 데이터베이스의 논리적 구조가 변경되어도 응용 프로그램에는 영향이 없어야 한다.

3층 스키마의 구조 : 1. 외부 스키마: 사용자나 애플리케이션 입장에서 필요로 하는 데이터베이스 구조.
			        외부스키마는 여러 사용자가 공유하여 사용할 수 있다.
			        데이터베이스의 여러 관점에 대하여 기술한다. (하나의 DB에 외부 스키마는 여러개가 존재할 수 있다.)	

		 2. 개념 스키마 : 기업이나 조직체 입장에서 바라본 데이터베이스 구조
			         개체 관계 속성을 정의한다. (전체 데이터베이스의 규칙과 구조를 표현한다.)
			         통합 데이터베이스 구조 (설계자 관점)
		3. 내부 스키마 : 물리적 저장장치의 입장에서 바라본 데이터베이스 구조(개발자 관점)
			         데이터의 저장구조, 필드 정의 , 레코드 정의 인덱스등을 의미한다.

엔터티 - 엔터티는 업무에서 사용되는 데이터(집합)이며, 저장되고 관리되어야 한다.
           엔터티는 개념, 사건, 장소 등의 명사이다.

엔터티의 특징 - 1. 식별자 - 엔터티는 유일하게 튜플을 식별할 수 있는 식별자가 존재해야한다.
	        2. 인스턴스 집합 - 2개 이상의 인스턴스가 존재하여야 한다. 
	        3. 속성 - 엔터티는 반드시 두 개 이상의 속성을 지녀야한다.
	        4. 관계 - 엔터티는 반드시 다른 엔터티와 반드시 하나이상의 관계를 지녀야 한다.
	        5. 업무 - 엔터티는 업무에서 관리되어야 할 집합이다.


엔터티의 종류 - 1. 유무형에 따른 유형
		- 유형 엔터티 : 업무에서 돌출되며 지속적으로 사용되는 엔터티(물리적 형태가 있는 엔터티)
		- 개념 엔터티 : 물리적인 형태가 존재하지않고 개념적으로 사용되는 엔터티이다.
		- 사건 엔터티 : 비지니스 프로세스를 실행하면서 돌출되는 엔터티이다.
	       -2. 발생 시점에 따른 유형
		기본 엔터티 : 키 엔터티라고도 불리며, 다른 엔터티로 부터 영향을 받지 않고 독립적으로 생성되는 엔터티이다.
		중심 엔터티 : 기본엔터티와 행위엔터티 중간에 존재하는 엔터티로 기본엔터티에 의해 생성되며 행위 엔터티를 생성한다.
		행위 엔터티 :2개 이상의 엔터티로 부터 발생하는 엔터티이다.


 
속성 - 업무에서 관리되는 데이터이다.
       - 속성은 하나의 값만을 가진다.
       - 주식별자에 의하여 함수적으로 종속된다.

속성의 종류 - 1.분해 여부에 따른 속성의 종류 
		단일속성 : 하나의 의미로 구성된 속성
		복합속성 : 여러개의 의미가 있는 것 (복합키 : 기본키를 구성하는 여러 속성)
		다중값 속성 : 속성이 여러개의 값을 가지는 것으로 다중값 속성은 엔터티로 분해된다.
	   - 2. 특성에 따른 속성의 종류
		기본 속성 : 업무 프로세스로 부터 돌출되는 본래의 속성(회원이름, 계좌번호 등)
		설계 속성 : 데이터 모델링 과정에서 생성되는 속성 (유일한 값을 부여한다. ex: 상품코드, 지점코드 등)
		파생 속성 : 다른 속성들에 의해서 만들어진 속성(ex : 평균값, 분산값 등)

관계 - 엔터티와 엔터티 사이에서의 관련성을 말한다.
      
관계의 종류 - 1. 존재 관계: 엔터티간의 상태를 말한다. (고객이 은행에 회원가입을 할 경우. 은행에서는 해당 고객을 담당할 관리지점을 부여한다.
					        [즉. 관리지점에 고객이 소속되어있는 상태]에 대한 관련성을 기술하는 것이 존재 관계이다. )
	     2. 행위 관계:  엔터티간의 어떤 행위가 발생하는지에 대하여 관련성을 기술하면 행위관계이다.





관계 차수 - 두 개의 엔터티 간에 관계에 참여하는 수를 의미한다.
	완전 1:1 관계 : 엔터티에서 각각의 튜플이, 관계를 맺은 엔터티 내에 존재하는 단 하나의 튜플과 관계를 가지고 있는 경우
	선택적 1:1 관계: 하나의 엔터티에 관계되는 엔터티가 하나가 존재하거나 없을 수 있다.

	1:N 관계: 엔터티의 하나의 튜플이 관계를 맺고있는 엔터티의 여러 행과 관계를 맺고 있는 경우를 말한다.
	N:M 관계: 관계를 맺고 있는 엔터티들의 각각의 튜플이 서로 다수의 튜플과 관계를 맺고 있는 경우. 

식별관계 - 기본키가 참조릴레이션의 기본키로써 존재하는 경우
비식별관계 - 기본키가 참조릴레이션의 기본키가 아닌 경우

(식별자)
키의 종류 : 1. 기본키: 후보키중에서 유일하게 선택된 키로써, 유일성과 최소성을 만족하며, 엔터티를 대표할 수 있는 속성을 말한다. 
	  2. 슈퍼키: 엔터티 내에 존재하는 속성들로 구성된 속성 집합으로 유일성은 만족하나 최소성은 만족하지 않는 키이다.
	  3. 후보키: 유일성과 최소성을 만족하는 속성들의 집합으로 구성된 키이다.
	  4. 외래키: 참조할 릴레이션의 기본키 속성
	  5. 대체키 : 후보키에서 기본키를 제외한 속성들을 말하낟.

식별자의 종류 - 대표성 여부에 따른 식별자 종류 : 1. 주식별자 - 엔터티를 대표하는 속성
					  2. 보조식별자 - 유일성과 최소성은 만족하나 엔터티를 대표하지 못하는 속성
	      - 생성여부에 따른 식별자 종류 : 1. 내부식별자 - 엔터티 내에서 스스로 생성되는 식별자.
		 		          2. 외부식별자 - 엔터티와의 관계로인해 만들어지는 식별자.
